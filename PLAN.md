# Guesswhere — Build Plan

A satellite-imagery city-guessing game. Player picks a population tier, gets 10 cities,
identifies each from imagery alone using a minimap for reference.

**The problem this project exists to solve:** the original game's minimap and answer key come
from two unrelated databases, so they disagree — alternate spellings, missing places, and
wrong names (especially in China). This build makes disagreement *structurally impossible*.

---

## Core architectural decision

> **The answer key is extracted from the minimap tiles themselves.**

The minimap is self-hosted vector tiles. The answer key is generated by reading the place
labels out of that same tile file. There is only one name per city, used for both rendering
and grading — so there is nothing to reconcile and nothing to drift.

**Invariant: if a city's label is not rendered on the minimap, it is not in the answer key.**
Never add cities to the pool from an outside source.

**Invariant: tiles and answer key ship as one versioned artifact.** If tiles are rebuilt, the
answer key is rebuilt in the same step. Two independent pipelines = the original bug, reintroduced.

### Why the main imagery needs no alignment

Mapbox raster tiles are coordinate-addressed (`z/x/y` derived from lat/lng). We send
coordinates, we get pixels. Mapbox never sees a name and has no opinion to disagree with.

## Settled decisions

| Question | Decision | Rationale |
|---|---|---|
| Main imagery | **Mapbox Satellite via GL JS** | ~0.5 m/px globally; verified legible field text |
| Billing model | **GL JS map loads, NOT raster tile API** | 1 load/game vs ~3,000 tile requests. Critical. |
| Do not cache Mapbox tiles | ToS prohibits it, and map-load billing makes it pointless | |
| Minimap | **Self-hosted Protomaps PMTiles**, capped ~z13 | Free, offline, and enables the answer-key inversion |
| Main view layers | **Pure imagery only** — no vectors, no labels, no overlays | |
| Minimap layers | map / elevation / (satellite deferred) | |
| Hosting | **Cloudflare** — R2 + Workers + Durable Objects | R2 zero egress; Fly free tier is dead |
| Grading | **Server-side.** Answer never sent to client | Same code cost; keeps leaderboards honest |
| Population semantics | **GHSL**, not administrative population | Makes "50,000" mean the same thing in Shandong and Saxony |

### Verified facts (checked, not assumed)

- Esri World Imagery max zoom varies z17–z19 by region; ~1 in 5 sampled cities capped near 1 m/px. **Rejected in favor of Mapbox.**
- Mapbox free tier: 50,000 map loads/month, then $5/1,000. At ~1 load/game that's ~50k free games/month.
- Mapbox Product Terms contain **no prohibition on games**. (Restrictions are gambling/economic-harm and competing-with-Mapbox.)
- Fly.io retired free allowances for new accounts in 2024.
- `worldcities.csv` (SimpleMaps, 50,250 rows) has **no alias column** — demoted to cross-check only, not a source of truth.

## Phases

### Phase 1 — City database → **Sonnet** ✅ DONE

Built in `etl/`; output `etl/cities.sqlite` (3.6 MB). **11,000 playable cities across
190 countries** (96.3% of GHSL's 11,422 urban centres); 422 quarantined. China's rate
matches the global average, which is the point of the whole project.

Notes for anyone re-running it:
- GHSL's `GC_UCN_LIS_2025` is double-encoded UTF-8 in 754 rows (`text-repair.js` fixes it),
  and it lists **absorbed neighbouring settlements**, not alternate spellings — it must
  never become aliases (Kolkata's list contains "Howrah").
- Non-Latin-script countries need `name:en` from the tile, not just `name`.
- GHSL's main name must outrank its absorbed list when scoring, or a satellite city
  hijacks the slot (Shenzhen was winning Guangzhou's).

Build the ETL (Node + DuckDB; note: system `python` is only the Microsoft Store stub).

1. Acquire Protomaps planet PMTiles, cap to ~z13.
2. Extract all `place` layer labels → `(name, lat, lng, country, min_render_zoom)`.
3. Join GHSL by proximity → population.
4. Join GeoNames `alternateNamesV2` → alias array.
5. Cross-check against `worldcities.csv`; report disagreements, don't auto-resolve.
6. **Quarantine** anything with no label, no population, or a suspicious join.

**Output:** `cities.sqlite`
`id, canonical_name, aliases[], lat, lng, country, iso2, pop_ghsl, min_render_zoom, tier, quarantined`

**Acceptance:** every non-quarantined city renders a visible minimap label at some
reachable zoom. Spot-check 20 across varied regions, including China.

### Phase 2 — Matching engine → **Opus** ✅ DONE

Lives in `matching/`. Server-side; the answer never reaches the client.

- `normalize.js` — generates a *set* of variant forms per name rather than one
  canonical form, so `广州市`→`广州`, `Saint`/`St`, and Arabic `Al-`/`Ad-` all work
  without a destructive rewrite. Script-preserving: local-script guesses are accepted.
- `match.js` — four tiers: own-canonical → foreign-canonical reject → own-alias →
  foreign-alias reject → bounded Damerau-Levenshtein.
- `name-index.js` — global index of all 11k cities' names. **This is what makes
  generous typo tolerance safe.**
- `grader.js` — the service phase 3/4 calls.

**Key design decision.** Real city pairs sit ONE edit apart: Kanpur/Kannur, Pune/Puno,
Vienna/Vienne. No distance threshold can separate those from ordinary typos. So the
fuzzy budget stays generous and correctness comes from a different rule: *a guess that
exactly names another real city is a wrong answer, not a misspelling.* Homonyms are
unaffected, since each city is canonical for itself.

Damerau (transposition = 1 edit, not 2) matters more than expected — it took typo
tolerance from 79.6% to 92.9%, since transposition is the most common human typo.

Measured on the real 11,000-city corpus (`node validate.js`):

| Metric | Result |
|---|---|
| Self-acceptance | 11,000/11,000 |
| False-accept rate | **0.0028%** (1/35,998 — and that one is a genuine homonym) |
| Typo tolerance | 92.9% of single-edit typos forgiven |
| Throughput | ~950k grades/sec |

Remaining typo misses are names ≤4 chars, where the budget is deliberately 0.

Tests: `node --test` in `matching/` (23 tests).

### Phase 3 — Game client → **Sonnet** ✅ CORE LOOP DONE

Built in `web/` (Next.js 16 App Router + Turbopack). Verified end-to-end through the
actual UI: new game → 10 cities, no repeat countries → guess → green/read-only/counter
increments → matches PLAN.md spec exactly.

**Structure:**
- `lib/server/grader.ts` -- wraps `matching/` (wired in as a `file:../matching` local
  npm dependency, not a raw relative import -- Turbopack refuses to resolve paths
  outside its project root otherwise; see `turbopack.root` in `next.config.ts`)
- `lib/server/gameLogic.ts` / `gameStore.ts` -- round selection (population-band
  widening search, no-repeat-country), in-memory session store
- `app/api/game/*` -- new/guess/reveal/state. Answer never sent to client (grade
  server-side, only `{correct, canonicalName}` on success crosses the wire)
- `app/api/tiles/[file]/route.ts` -- HTTP Range server for the local 35GB planet
  PMTiles file (dev/small-scale; R2 is the upgrade path, not built yet)
- `components/MainMap.tsx` -- Mapbox GL JS satellite view, one map instance reused
  across all 10 rounds (map-load billing, not tile billing), wide/pinpoint/rubber-band
  pan mechanics
- `components/MiniMap.tsx` -- MapLibre + pmtiles protocol + `protomaps-themes-base`,
  reading the SAME tiles that are the answer key. Elevation layer via AWS's public
  Terrarium tiles (phase 1's dedicated elevation source was deferred; this is a free
  stand-in, not a gap)
- `components/GameHeader.tsx` / `AnswerBox.tsx` -- exactly 2 buttons + 2 counters, per
  the corrected spec; green/locked/sound on correct, clear/shake/sound on wrong

**Known gotchas if this needs debugging later:**
- Turbopack's CJS interop breaks specifically on `require('node:sqlite')` reached
  through an external package (`Cannot find module 'node:sqlite': Unsupported
  external type Url for commonjs reference`) -- but a direct ESM `import { DatabaseSync }
  from 'node:sqlite'` works fine. `grader.ts` therefore does the sqlite access itself
  in TS and only `require()`s the pure-JS matching logic (`match.js`, `name-index.js`,
  which never touch sqlite) from the shared package.
- Next 16 dev mode promotes any client-side `console.error()` to a full-screen
  blocking overlay. Use `console.warn` for expected/recoverable conditions (missing
  token, map errors) -- reserve `console.error` for things that should actually halt
  attention.
- Not yet verified by me: the minimap actually painting tiles on screen. This sandbox's
  browser tab reports `document.hidden = true` (a Page Visibility API artifact of the
  preview harness), which throttles the `requestAnimationFrame` loop both Mapbox GL JS
  and MapLibre GL JS depend on to reach their `load`/`styledata` events -- so the
  render step itself can't be confirmed from here. Everything upstream of rendering
  IS verified: the tile endpoint serves correct bytes (checked via direct fetch with
  a Range header), `protomaps-themes-base` produces schema-correct layers referencing
  the exact source-layer names phase 1 confirmed (`earth`, `roads`, `places`, etc.),
  and the whole game loop works through the real UI. **Open a real browser tab to
  `localhost:3000` to do the final visual check** -- add a Mapbox token to
  `web/.env.local` first.

**Not yet built:** pinpoint-button click-through test (logic is in place, untested
live), sounds (implemented via synthesized Web Audio tones, unverified audibly),
Mapbox ToS ordered ahead of ship. Deferred per plan: leaderboards (phase 4 -- still
not started; multiplayer/phase 5 is done, see below and `CLAUDE.md`'s status section).

### Phase 3 follow-ups (post-launch feedback round)

- **Header is now 3 buttons** (Recenter, Reveal, **Report Round**), not 2 — superseding
  the earlier confirmed count. New Game / counters / buttons order: left / middle / right.
- **Report Round** (`lib/server/reportedCities.ts`, `app/api/game/[gameId]/report/route.ts`):
  for bad/unusable imagery (heavy cloud cover, etc.). Adds the city to a JSON-file-backed
  blocklist, persisted across restarts and **global across all future games**, then
  immediately swaps in a replacement (same population tier, no repeat country against the
  other 9). Solo-only — see the duels-mode spec below for the two-player version, which
  needs multiplayer infra that doesn't exist yet.
- **Population semantics corrected: it's a MINIMUM, not a target band.** "2,000,000" means
  every city is at least that populous, no upper bound at all -- Jakarta is a legal answer.
  An earlier version (mistakenly, before this correction) treated the input as the center of
  a tolerance band, which silently capped how large a city could appear. That version also
  went through a band-tightening pass first (±2x → ±15%) chasing what looked like the same
  symptom, before the actual model was corrected here; both fixes are captured for the
  record, but only the floor model is what's shipped. `selectRound`/`pickReplacementCity`
  now just filter `pop_ghsl >= floor`; verified against the real corpus that countries stay
  comfortably available (45 at a 5M floor, 23 at 10M) with graceful repeat-country fallback
  only above ~15M, where just 9 countries qualify at all.
- **The Colachel data point, investigated**: real population is ~20-31k by any normal
  source, not the ~1M GHSL reports -- but this is not a simple bug to fix. GHSL's row shows
  why: it's a 325 km² urban centre merging six separate towns (Colachel, Parassala,
  Thengapattinam, Kuzhithura, Marthandam, Padmanabhapuram) under whichever name our tile-
  matching landed on. Checked how common this is (400 cities show a >=5x, same-name
  population disagreement against worldcities.csv) and found the signal is NOT reliable
  enough to auto-correct: some of those 400 are genuine merge artifacts like Colachel, but
  others are cases where GHSL is MORE correct (Ankara, New Delhi, Sri Jayewardenepura Kotte
  -- real metro-scale cities where worldcities.csv's figure is a stale/narrow administrative
  count), and a meaningful fraction of worldcities.csv's own rows are visibly broken
  (population values of 1, 98, 130 -- not usable as ground truth either way). Building an
  automated classifier on data this mixed risks doing more damage than the original problem.
  **Report Round is the intended mitigation** for cases like this as they're actually
  encountered in play, rather than a best-effort automated pass across the whole corpus.
- **Wide-view dimensions corrected**: 42.8 km × 18.2 km (was 27 × 12, a rough guess from the
  initial description). Directly measured against the original game: 26.59mi E/W × 11.31mi N/S.
- **Minimap contrast**: stock `namedTheme('light')` renders forest/scrub/sand/earth within a
  few RGB points of each other, and building footprints the same color as the canvas
  background (i.e. invisible). Switched to `layersWithPartialCustomTheme` with overrides in
  `components/MiniMap.tsx` — green vegetation, warm tan dry land, visible grey building
  footprints — so population-center shape reads at a glance instead of a flat wash.
- **Minimap starting view**: now a fixed neutral world view (`[10, 15]` at zoom 1.5),
  identical every round — not centered anywhere near the answer. An earlier version offset
  the start 30-70km from the answer in a per-city direction; the user asked for the simpler
  "always starts at the world" behavior instead, which is what's shipped.

#### Duels/ranked mode — BUILT (2026-07-22), diverges from the original spec below

What actually shipped (see `CLAUDE.md`'s status section + `web/lib/server/duelLogic.ts`):
host-configurable per-round timer (auto-skips the round for everyone on expiry),
"first to N correct" instead of fixed 10 rounds, live leaderboard, join via a 4-char
code (not a link). Real-time sync is polling (`GET /api/duel/[lobbyId]/state`,
~750ms) against a server-authoritative lobby with absolute deadline timestamps +
tick-on-read, not Cloudflare Durable Objects -- the app ended up hosted on Railway
(a persistent Node process) rather than Workers, so a Durable Object per room was
never the natural fit it would have been under the original Cloudflare-hosting plan.

Two things from the original spec below were **deliberately not built**, because the
actual ask when phase 5 was built didn't include them: the mutual skip-vote (Report
Round exists in solo but not duels at all -- no reveal/skip mechanic in duels v1), and
any accounts/persistent identity (duels' `playerId` is a throwaway per-lobby
`localStorage` value). Both are still open for whenever they're actually requested.

Original spec follows, kept for the historical record per this doc's own convention:

User requirements for competitive multiplayer, captured before phase 5 was built:
- Per-round timer (~5 min); if neither player answers in time, auto-skip to the next round.
- A **separate mutual skip-vote**, distinct from Report Round: either player can vote to
  skip a round for reasons other than bad imagery, but it only actually skips once **both**
  players have voted. Report Round stays unilateral and immediate (bad imagery is bad
  imagery regardless of the other player's opinion); a bare "skip" needs consensus, since
  the round itself is perfectly fine — just less preferred by one player.
- Ranked games are **not** fixed at 10 rounds — something like "first to 5 correct."

### Phase 4 — Accounts + leaderboards — BUILT (2026-07-22)

The one-line spec below ("server-authoritative timing, per-tier speed leaderboards,
revealed rounds don't score") all shipped. Details and rationale live in `CLAUDE.md`'s
status section; the decisions worth recording here are the ones where the answer wasn't
obvious from the spec:

- **"How long you spent on a round" is accumulated ACTIVE time**, not first-arrival →
  solve. The literal reading was the original ask, but because you can paginate freely
  between the 10 slides, it double-counts: Toronto's clock would keep running while
  you're looking at Jakarta, and the round times would sum to far more than the game
  actually took. Active time means exactly one round accrues at a time, so the
  breakdown and the leaderboard number can never disagree. Confirmed with the user
  before building.
- **Only the four preset tiers have boards**, matched exactly rather than bucketed.
  Population is a *minimum*, so a higher tier = bigger, more recognizable cities = an
  easier game; folding a custom 137k run into the 100k board would let anyone hand-pick
  an easier setting and rank against people who played the real one.
- **Cloned ("Share Cities") sets never rank.** Found while wiring clone ownership: a
  clone is by construction a set someone has already played, so without this you could
  finish a game, clone it, and speedrun the memorized answers to the top of every board.
  The cost — a friend seeing the set fresh can't rank either — is accepted, because the
  server genuinely cannot distinguish the two cases.
- **Email is optional, and reset requires a *verified* address.** Optional so a mail
  problem never gates play; verification-required so signing up with someone else's
  address isn't an account-takeover path.
- Accounts/leaderboards persist because `GAME_DB_PATH` sits on the Railway volume. The
  prune sweep added alongside them is safe for leaderboards **by construction**, since
  `game_results` are self-contained snapshots rather than references into `games`.
